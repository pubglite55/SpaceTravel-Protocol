## 蓝牙验证机制分析

### 可能的验证方式

1. **RFCOMM 连接验证**
   - 耳机在 SDP 查询时检查连接方的 APP 签名
   - 通过 Android 的 `BluetoothDevice.createRfcommSocketToServiceRecord()` 时验证

2. **GAIA 协议认证**
   - 连接建立后，耳机发送认证挑战
   - APP 需要用私钥签名响应

3. **包名验证**
   - 耳机可能只接受特定包名的 APP 连接
   - 需要匹配 `com.moondroplab.moondrop.moondrop_app`

### 验证流程推测

```
┌─────────────┐                    ┌─────────────┐
│   手机 APP  │                    │    耳机     │
└──────┬──────┘                    └──────┬──────┘
```

## 绕过方案评估

### 方案1: 使用官方签名打包 (推荐)

**步骤:**
1. 获取官方签名密钥 (需要私钥)
2. 用官方密钥签名我们的 APK
3. 修改包名为 `com.moondroplab.moondrop.moondrop_app`

**问题:** 无法获取私钥

### 方案2: Hook 官方 APP

**步骤:**
1. 用 Frida hook 官方 APP 的蓝牙通信
2. 拦截 RFCOMM 连接建立过程
3. 提取认证握手数据
4. 重放认证数据

**可行性:** 中等

### 方案3: 修改系统蓝牙配置

**步骤:**
1. 修改 Android 蓝牙配置
2. 禁用签名验证
3. 允许所有 APP 连接

**问题:** 需要 root 权限，可能影响其他蓝牙功能

### 方案4: 重放攻击

**步骤:**
1. 用官方 APP 建立连接
2. 记录认证握手数据
3. 用我们的 APP 重放相同的数据

**可行性:** 低 (可能有随机数/nonce)

## 推荐方案

**最佳方案: Frida Hook + 中间人**

1. Hook 官方 APP 的 `BluetoothSocket.connect()`
2. 拦截发送和接收的数据
3. 分析认证握手协议
4. 在我们的 APP 中重放认证数据

## 证书文件

APK 路径: 从设备提取的官方 APK

提取签名:
```bash
# 提取证书
keytool -printcert -jarfile MOONDROP_official.apk

# 或使用 apksigner
apksigner verify --print-certs MOONDROP_official.apk
```
